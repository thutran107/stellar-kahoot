★ Field Notes · Real-Time Games

Lessons from Hardening a
Real-Time Multiplayer Game

Notes from making a Kahoot-style game (Socket.io + React, in-memory server state) survive real use. Mostly about failure modes, identity, testing, and knowing your actual scale before you optimize.

Most of this is not about features. It's about what breaks when real phones, flaky wifi, and a room full of players hit code that worked fine on your laptop.

1

The client is the constraint, not the server

A player on iOS Safari got “A problem repeatedly occurred” mid-game and was dropped. That message isn't a JavaScript error — it's the browser killing the tab's render process for using too much memory/GPU, then crash-looping on reload.

The cause was the UI, not the network: the player screen ran ~70 continuously animating, GPU-composited layers (large blur() filters with mix-blend-screen, plus several requestAnimationFrame loops) for the whole game. Fine on a laptop, fatal on a loaded phone.

Lessons
  • In a live game, every player runs your heaviest UI on the weakest device in the room. Design for the worst phone.
  • Animated blur() + mix-blend-screen every frame is the most expensive thing you can ask a mobile GPU to do. Gate heavy effects behind a mobile / prefers-reduced-motion check; ship a static fallback.
  • Distinguish a render-process crash (white tab, OOM) from a JS exception (overlay, stack trace) — totally different causes.
2

Three kinds of testing — they don't substitute for each other

Run a load test, see green, call the game “ready” — that's a trap. Each layer answers a different question:

LayerAnswersBlind to
Load test
(headless socket clients)		Can the server hold N connections and deliver answers?Anything the browser renders. It never draws a pixel.
Unit / integration
(in-process)		Is the game logic correct (scoring, state machine)?Real network, real devices.
Device test
(real phone / throttled)		Does it actually work and survive on real hardware?Nothing else catches the client OOM.

The mobile crash above could never be caught by a load test — load tests use headless clients with no DOM. We nearly reached for “run more load tests” to validate a client crash. Wrong layer. Name what each test proves; don't let green in one layer imply safety in another.

3

Make real-time code testable, or you'll never change it safely

The game logic started sealed inside one big startServer() that also booted the web server and listened on a port. You couldn't reach scoring or the socket handlers without starting the whole thing — so the part most likely to break had no tests.

The fix was a behaviour-preserving extraction: move session state and all socket handlers into an exported registerGameHandlers(io, options). Now a test can spin up a throwaway Socket.io server on an ephemeral port, connect real clients, drive a real game (host → join → answer → results), and assert on the broadcast payloads.

Lessons
  • Test real-time logic through the protocol (emit events, assert on broadcasts), not by poking internal state — keeps tests honest about what clients see.
  • Mock only the truly external dependency (the database). Run the real socket server in-process.
  • Make timeouts injectable (registerGameHandlers(io, { hostGraceMs })) so a test uses 150ms, not 30s. Untestable timing is untested timing.
  • This extraction was the highest-leverage move — every later fix became verifiable the moment the logic was importable. Do it before, not after.
4

The “answer storm”: quadratic broadcasts — and why we did not fix it

The server broadcast full game state to every socket on every submitted answer. With N players answering at once that's ~N broadcasts per question, each an O(N) payload to N clients: O(N²) messages, O(N³) bytes per question. We measured instead of guessing:

PlayersBroadcasts / questionPayload pushedEvent-loop stall
502,65222.8 MB90 ms
10010,302173.7 MB480 ms

Doubling players (50→100) gave 3.9× the broadcasts and 7.6× the bytes. The scary metric isn't megabytes — it's the event-loop stall: at 100 players the server froze ~half a second per question, unable to accept joins or answer health checks.

The decision: we did not fix it. The real target was ~40 players, where the stall is ~50ms (imperceptible), confirmed by a 40-player production run. Fixing it would have been engineering for a scale that didn't exist.

Lessons
  • Measure before you optimize — we almost shipped a fix for a non-problem.
  • Event-loop stall, not bandwidth, predicts a bad live game.
  • “Broadcast full state on every mutation” is a fine default that quietly goes quadratic. Know where your cliff is.
  • Knowing your real scale is a feature — it told us to skip a whole project.
5

Identity and reconnection: the genuinely hard part

Phones drop, wifi blinks, tabs crash — constantly, in a live game. The default setup made it catastrophic because players and the host were keyed by the ephemeral socket.id. A reconnect is a new socket = a new identity, so a dropped player was deleted, and the host dropping ended the game for everyone.

Stable identity beats socket id

The fix: a client-minted token (a UUID in localStorage) sent on join/host and re-sent on reconnect. The server keys players/host by that token, not the socket. A reload or blip reattaches to the same identity with the same score.

Disconnect ≠ “gone” — use a grace window

For the host, a 30s grace timer: a disconnect starts a countdown instead of ending the room; reconnect cancels it. Same idea for players (keep them listed as “reconnecting,” don't delete).

The edge cases are where the real work is

80% of a reconnect feature
  • Recycled identifiers. Short PINs get reused. A stale localStorage entry must not walk a returning user into a different game sharing the PIN — match on the unguessable token, clear only the exact session you tried to resume.
  • Reload vs. blip. A transient reconnect keeps in-memory state; a full reload wipes it and re-runs mount logic, racing with reconnection. An isResuming flag stops a reloading host from auto-hosting a duplicate.
  • localStorage can throw. iOS private mode throws on write — guard every access and degrade to “no resume” instead of crashing the join.
  • Re-emit on every connect. Handle reload and transient reconnect with one path: on every connect, if a stored session exists, attempt resume.

Trust model

The token is client-generated and trusted: possession is identity. A 122-bit UUID is unguessable, so it's fine for a party game — name that assumption so nobody mistakes it for auth.

6

In-memory state is fast and fragile — decide what you won't cover

All game state lived in plain server-side Maps. Great for latency, but:

  • A server restart wipes every live game. Our reconnection work survives a client crash but not a server restart — we accepted that and wrote it down rather than building DB-backed live-state recovery (a much larger project for a party game).
  • On a free hosting tier the instance sleeps when idle, so the first players can hit a 30–60s cold boot. A self-ping doesn't help (the process is stopped); only an external warmer does.
Lesson
  • For in-memory real-time servers, explicitly decide and document what you do not recover from. “Survives client crashes, not server restarts” is a fine, honest boundary. Silent gaps are the dangerous ones.
7

Production is a pile of small decisions, not features

The interesting work was rarely “build X.” It was choosing well: how long a host can be gone before the room gives up (30s); what other players see while someone's out (“reconnecting,” not vanished); whether reconnection is silent or manual (silent); how much we cover for a server restart (nothing, documented).

Principles that held up
  • Do the complete thing when it's cheap, flag the thing that isn't. A clean reconnect across blips and reloads was worth finishing; DB-backed recovery was correctly flagged out of scope.
  • YAGNI against your real scale. The answer-storm fix was the textbook “right” optimization and the wrong thing to build for 40 players.
  • Be honest about what's verified. Server logic had tests; the client reconnection path type-checked and built but wasn't auto-tested — so it was called out as needing a real-device check, not quietly marked “done.”
8

Process notes that paid off

  • Tag a known-good commit before risky work. A named anchor makes “undo all of this” one line instead of archaeology.
  • Refactor behind tests, then change behavior. Extract → baseline test → modify. The baseline catches a refactor breaking scoring instantly.
  • Test-first shines on state machines. Reconnection and scoring are exactly the silent-corruption-prone logic where a failing-then-passing test earns its keep.
  • Small, independent PRs keep each change revertable on its own.

TL;DR

  1. The client (mobile GPU/memory) is your real constraint; load tests are blind to it.
  2. Extract your real-time logic so it's importable and testable through the protocol.
  3. Measure scaling before optimizing; event-loop stall predicts a bad live game; know your cliff and ignore it if you're below it.
  4. Key players/host by a stable token, not the socket id; treat disconnect as “maybe coming back” with a grace window; the edge cases are the actual work.
  5. In-memory is fast and fragile; decide and document what you do not recover from.
  6. Most of production is small, well-reasoned decisions about failure modes.
